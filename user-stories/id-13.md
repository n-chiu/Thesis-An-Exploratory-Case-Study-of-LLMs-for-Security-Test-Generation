# (13) Cryptographic Failures

## Title: Entropy in key generation

## ID: 13

## User Story:
As a Software company Customer, I need a suitable level of entropy in generating keys to prevent attacks.

## Acceptance Criteria:

Verify that all random numbers, random file names, random GUIDs, and random strings are generated using the cryptographic module’s approved random number generator when these random values are intended to be not guessable by an attacker.

## Edge Cases:

Use of insecure random generators.

Predictable seeds.

Reuse of random values (e.g. duplicate tokens or GUID collisions).

Blocking entropy sources causing performance issues.

Fallback to weaker RNG when secure RNG fails.

## Constraints:

Only use cryptographically secure RNGs (CSPRNG).

Must rely on OS-provided entropy sources.

No custom RNG implementations.

Random values must be sufficiently long and unpredictable.

Systems must ensure entropy availability.

## Vulnerability Category:

Cryptographic Failures

## CWEs:

Cryptographic Failures: 326, 327, 331, 332, 334, 335, 336, 337, 338, 340, 342.
