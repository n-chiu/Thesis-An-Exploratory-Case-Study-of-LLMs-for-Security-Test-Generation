# (6) Cryptographic Failures

## Title: Encrypt credentials

## ID: 06

## User story:
As a Software company Customer, I want my account credentials to be created, stored and transported securely so that they can’t be guessed, intercepted or reused by an attacker.

## Acceptance Criteria:

Verify that account passwords are one way hashed with a salt, and there is sufficient work factor to defeat brute force and password hash recovery attacks.

Verify that credentials are transported using a suitable encrypted link and that all pages/functions that require a user to enter credentials are done so using an encrypted link.

## Edge Cases:

Weak hashing algorithms still in legacy systems.

Missing salt or reused salts across users.

TLS downgrade attacks (man-in-the-middle).

Credentials exposed in logs or error messages.

API clients sending credentials over HTTP instead of HTTPS.

Improper certificate validation.

## Constraints:

Use strong hashing (bcrypt, Argon2, PBKDF2 with high work factor).

Unique salt per password.

Enforce HTTPS everywhere, no HTTP fallback.

Secure certificate validation required.

Secrets must never be logged or cached.

Vulnerability Category:

Cryptographic Failures

## CWEs:

Cryptographic failures: 261, 296, 319, 321, 322, 323, 324, 325, 326, 327, 328, 334, 340, 347, 523, 759, 760, 916, 1240, 1241 (almost all).
