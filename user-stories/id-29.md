# (29) Injection

## Title: Input validation

## ID: 29

## User Story:
As a Software company Customer, I want the web service to validate input.

## Acceptance Criteria:

The same encoding style is used between the client and the server.

Confirm that XML or JSON schema is in place and verified before accepting input.

Ensure that all input is limited to an appropriate size limit.

A REST service explicitly checks the incoming Content-Type to be the expected one, such as application/xml or application/json.

## Edge Cases:

Malformed JSON/XML with correct Content-Type.

Wrong charset or inconsistent encoding between client and server.

Missing required fields, wrong types, or unexpected fields.

Oversized payloads or deep nesting.

XML external entities or entity expansion attempts.

Correct header but mismatched body format.

Boundary-size inputs at and around configured limits.

Schema-valid but semantically invalid business values.

## Constraints:

Server-side validation is mandatory for all input.

Only approved content types and UTF-8 encoding are accepted.

XML/JSON schema validation must occur before processing.

Strict request, field, depth, and array-size limits must be enforced.

Validation errors must fail safely and not leak internals.

Validation rules must be consistent across all endpoints and API versions.

## Vulnerability Category:

Injection

## CWEs:

Injection: 76, 79, 80, 91, 112, 116
