# (19) Software Supply Chain Failures + Insecure Design

## Title: Isolate components to avoid contamination

## ID: 19

## User Story:
As a Software company Customer, if an application component is vulnerable, I want the malicious activity against it to have minimal or no impact on the rest of the application.

## Acceptance Criteria:

All malicious activity is adequately sandboxed, containerized or isolated to delay and deter attackers from attacking other applications.

## Edge Cases:

Compromised service accessing shared database or storage.

Lateral movement between containers due to weak network segmentation.

Shared credentials across services enabling escalation.

Resource exhaustion (DoS) affecting other tenants/services.

Misconfigured container privileges (e.g. running as root).

Side-channel attacks between co-located workloads.

## Constraints:

Strong isolation via containers, VMs, or sandboxing.

Network segmentation between services (zero-trust principles).

Least privilege for service accounts and inter-service communication.

Resource limits enforced per component.

No shared secrets across isolated components.

## Vulnerability Category:

Software Supply Chain Failures + Insecure Design

## CWEs:

Software Supply Chain Failures: 1104, 1329, 1357, 1395.

Insecure Design: 501, 642, 653
