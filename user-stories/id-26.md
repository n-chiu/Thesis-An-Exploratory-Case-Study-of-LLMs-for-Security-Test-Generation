# (26) Broken Access Control

## Title: Principle of least privilege

## ID: 26

## User Story:
As a Software company Customer, I want the mobile client to observe the principle of "least privilege”.

## Acceptance Criteria:

The application requests minimal permissions for required functionality and resources.

## Edge Cases:

App requesting permissions not directly tied to features (e.g. contacts, location).

OS permission changes after app updates.

Users denying permissions causing app failure.

Background services accessing restricted resources unnecessarily.

Third-party SDKs requesting excessive permissions.

Permission granted but not actively used.

## Constraints:

Request only permissions strictly required for functionality.

Use runtime permission requests (not all upfront).

Graceful degradation if permissions are denied.

Regular review of permissions and SDK usage.

Follow platform guidelines.

No hidden or undocumented permission usage.

Permissions must be justified and documented.

## Vulnerability Category:

Broken Access Control

## CWEs:

Broken Access Control: 276, 281, 282, 283, 732, 863
