# (9) Authentication Failures + Security Misconfiguration

## Title: Secure sessions against hijacking

## ID: 09

## User Story:
As a Software company Customer, application sessions must be unique and resistant to hijacking by a malicious actor.

## Acceptance Criteria:

All successful authentication and re- authentication generate a new session and session id.

The session id is never disclosed in URLs, error messages, or logs. This includes verifying that the application does not support URL rewriting of session cookies.

Validate that only session ids generated by the application framework are recognized as active by the application.

Confirm that session ids are sufficiently long, random and unique across the correct active session base.

Session ids stored in cookies must have their path set to an appropriately restrictive value for the application, and authentication session tokens additionally set the "HttpOnly" and "secure" attributes.

Ensure that the application limits the number of active concurrent sessions and that an active session list is displayed in the account profile or similar of each user. The user should be able to terminate any active session.

Verify the user is prompted with the option to terminate all other active sessions after a successful change password process.

## Edge Cases:

Session fixation attacks (session ID reused after login).

Session ID leakage via referrer headers.

Predictable session IDs.

Multiple concurrent sessions across devices.

Stolen cookies reused from another IP/device.

Improper cookie scope exposing sessions.

## Constraints:

Session IDs must be cryptographically random and high entropy.

Regenerate session ID after login and privilege changes.

Cookies must use HttpOnly, Secure and SameSite attributes.

No session IDs in URLs.

Limit concurrent sessions per user.

## Vulnerability Category:

Authentication Failures + Security Misconfiguration

## CWEs:

Authentication Failures: 384, 613

Security Misconfiguration: 614, 1004
