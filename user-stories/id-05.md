# (5) Authentication Failures

Title: Reset password safely

ID: 05

User story:
As a Software company Customer, I need the ability to securely change or reset my password without worrying that my account(s) can be hijacked by a malicious party.

Acceptance Criteria:

Verify all account identity authentication functions (such as update profile, forgot password, disabled / lost token, help desk or IVR) that might regain access to the account are at least as resistant to attack as the primary authentication mechanism.

At a minimum, verify that the changing password functionality includes the old password, the new password, and a password confirmation.

Verify that the forgotten password function and other recovery mechanisms do not reveal the current password and that the new password is not sent in clear text to the user.

Verify that forgotten password and other recovery paths use a TOTP or other soft token, mobile push, or other offline recovery mechanism. Use of a random value in an e-mail or SMS should be a last resort.

Edge Cases:

Expired or reused reset tokens.

Token interception via email compromise.

Race conditions allowing multiple resets.

Social engineering via help desk bypass.

User locked out after partial reset flow.

Constraints:

Reset tokens must be single-use, time-limited, and securely generated.

No disclosure of whether an account exists.

Multi Factor Authentication (MFA) required for sensitive account recovery where possible.

Passwords must never be sent via email/SMS in plaintext.

Rate limiting on reset attempts.

Audit logging for all recovery actions.

Vulnerability Category:

Authentication Failures

CWEs:

Authentication Failures: 620, 640
