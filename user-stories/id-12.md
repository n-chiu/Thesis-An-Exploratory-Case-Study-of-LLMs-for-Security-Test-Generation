# (12) Cryptographic Failures

## Title: Secure secret key management

## ID: 12

## User Story:
As a Software company Customer, I need access to application secret keys managed securely.

## Acceptance Criteria:

There is an explicit policy for how cryptographic keys are managed (e.g., generated, distributed, revoked, and expired).

Verify that this key lifecycle is properly enforced.

Ensure that consumers of cryptographic services do not have direct access to key material.

Isolate cryptographic processes, including master secrets and consider the use of a virtualized or physical hardware key vault (HSM)

## Edge Cases:

Keys hardcoded in source code or configuration files.

Developers or services accidentally logging key material.

Expired or revoked keys still being accepted.

Backup/restore processes exposing keys.

## Constraints:

All keys must be generated using approved cryptographic standards.

Centralized Key Management System (KMS) or HSM required.

Key rotation must be automated and non-disruptive.

Key material must never be exposed to application code or logs.

Full lifecycle management required: generation, storage, rotation, revocation, destruction.

## Vulnerability Category:

Cryptographic Failures

## CWEs:

Cryptographic Failures: 321, 322, 323, 324
