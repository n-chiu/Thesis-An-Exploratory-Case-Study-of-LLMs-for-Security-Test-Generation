# (14) Security Logging and Alerting Failures

## Title: Collecting logs without disclosure of sensitive information

## ID: 14

## User Story:
As a Software company Customer, I want the ability to collect logs in a standard format for a SIEM or other security tool, which contain information such as user and administrative access, but not sensitive information such as passwords or PII.

## Acceptance Criteria:

The application does not output error messages or stack traces containing sensitive data that could assist an attacker, including session id, software/framework versions and personal information.

The application does not log sensitive data as defined under local privacy laws or regulations, organizational sensitive data as defined by a risk assessment, or sensitive authentication data that could assist an attacker, including user’s session identifiers, passwords, hashes, or API tokens.

Security logging controls provide the ability to log success and particularly failure events that are identified as security relevant.

Each log event should include necessary information that would allow for a detailed investigation of the timeline when an event happens.

## Edge Cases:

Logs accidentally capturing PII.

Debug mode enabled in production exposing stack traces.

Correlation IDs missing, making event tracing impossible.

Log injection attacks.

Logs stored unencrypted or accessible publicly.

Third-party services logging sensitive data outside system control.

## Constraints:

Standardized log format (e.g. JSON, CEF) required for SIEM ingestion.

No sensitive data in logs.

Logs must include timestamps, user IDs, event types, and source IP.

Centralized logging system required.

Log retention policies must comply with regulations (e.g. GDPR).

Secure transport (TLS) for log forwarding.

## Vulnerability Category:

Security Logging and Alerting Failures

## CWEs:

Security Logging and Alerting Failures: 221, 223, 532, 778.
