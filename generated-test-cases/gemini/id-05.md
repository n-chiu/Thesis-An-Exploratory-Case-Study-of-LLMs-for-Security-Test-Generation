# ID 05

| ID | Test Case Title/Scenario | Preconditions | Test Steps | Expected Results | Environment Specification |
|----|--------------------------|---------------|------------|------------------|----------------------------|
| ID 1. Verify Password Reset Link Expiration | User has a valid account and has requested a password reset. | 1. Receive the reset email.<br>2. Wait for the defined timeout period (e.g., 24 hours).<br>3. Click the reset link. | The link should be invalid or expired. The user should be prompted to request a new link. | Web Staging / Chrome & Firefox |
| ID 2. Verify Single-Use Token Logic | User has requested a reset and received a link. | 1. Click the reset link.<br>2. Successfully change the password.<br>3. Attempt to use the same link again. | The link should be deactivated immediately after the first successful use. | Web Staging / Mobile Safari |
| ID 3. Invalidation of Existing Sessions | User is logged in on Device A and changes password on Device B. | 1. Log in on Device A.<br>2. Initiate and complete a password reset on Device B.<br>3. Attempt to navigate or refresh on Device A. | Device A should be automatically logged out (invalidated session) to prevent unauthorized access. | Multi-device Staging Environment |
| ID 4. Protection Against Token Brute-Forcing | A reset link exists for a user account. | 1. Attempt to guess the reset token/URL using a script.<br>2. Send 50+ rapid requests with slight variations to the reset endpoint. | The system should implement rate-limiting or account lockout after multiple failed attempts to use a reset token. | API Testing Environment (Postman/JMeter) |
| ID 5. Secure Password Requirements Validation | User is on the "New Password" entry screen. | 1. Enter a weak password (e.g., "123456").<br>2. Enter a password that matches the old one.<br>3. Click "Submit." | The system should reject the password, citing complexity requirements or the "cannot reuse old password" policy. | Web Staging / Chrome |
| ID 6. Email Privacy & Information Leakage | User enters an email that is not registered in the system. | 1. Navigate to the "Forgot Password" page.<br>2. Enter a non-existent email address.<br>3. Click "Reset." | The UI should show a generic message (e.g., "If an account exists, a link has been sent") to prevent "account enumeration" by attackers. | Web Staging / Chrome |
| ID 7. Man-in-the-Middle (MitM) Prevention | Reset page is accessed via an unencrypted connection. | 1. Attempt to access the password reset page using http:// instead of https://. | The server should automatically redirect to https:// or fail to load. HSTS headers should be present. | Production-Mirror Staging |
