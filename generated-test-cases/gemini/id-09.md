# ID 09

| Test Case Title/Scenario | Preconditions | Test Steps | Expected Results | Environment Specification |
|--------------------------|---------------|------------|------------------|----------------------------|
| Session ID Uniqueness and Entropy | User has a valid account and access to the login page. | 1. Capture session IDs for 50 consecutive logins.<br>2. Compare session IDs for duplicates.<br>3. Analyze the length and character set of the IDs. | 1. Every session ID must be unique.<br>2. IDs must be sufficiently long (min 128-bit) and complex to prevent brute-forcing. | Staging/UAT environment with traffic interceptor (e.g., Burp Suite). |
| Secure Cookie Attributes (HttpOnly & Secure) | Application is configured to use cookie-based sessions. | 1. Log into the application.<br>2. Inspect the session cookie via browser dev tools or proxy.<br>3. Verify the presence of HttpOnly and Secure flags. | 1. HttpOnly prevents XSS-based theft.<br>2. Secure ensures the cookie is only sent over HTTPS. | Web browser with DevTools; HTTPS enabled. |
| Session Invalidation on Logout | User is currently logged into an active session. | 1. Copy the active session ID/Cookie.<br>2. Click the "Logout" button.<br>3. Attempt to access a protected URL using the copied session ID. | 1. The server must invalidate the session on logout.<br>2. The user must be redirected to the login page with an "Unauthorized" status. | Standard Web Environment. |
| Session Fixation Resistance | Attacker can access the login page to retrieve a pre-login SID. | 1. Navigate to the login page and note the Session ID.<br>2. Log in with valid credentials.<br>3. Check the Session ID again. | 1. The Session ID must change immediately upon successful authentication (New ID assigned). | Proxy tool to monitor cookie changes. |
| Inactivity Timeout (Session Expiry) | User is logged in and idle. | 1. Log in and remain idle for a duration exceeding the defined timeout (e.g., 15 mins).<br>2. Attempt to refresh or click a link. | 1. The session must expire.<br>2. User must be prompted to log in again. | Server-side configuration set to short timeout for testing. |
| Session Hijacking via IP/User-Agent Change | User has an active, authenticated session. | 1. Capture the session cookie.<br>2. Attempt to use that cookie from a different IP address or browser (User-Agent). | 1. If strict binding is implemented, the server should reject the session and force a re-login. | VPN or different device/browser profile. |
