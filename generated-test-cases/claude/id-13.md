| TC-01 | Minimum entropy threshold met during key generation | Key generation module is deployed; CSPRNG (e.g. /dev/urandom) is available; entropy monitoring is enabled. ✔️ | (1) Trigger generation of a 256-bit symmetric key. (2) Capture entropy pool level before and after. (3) Analyse the key with NIST SP 800-22 tooling. ✔️ | Entropy estimate ≥ 7.99 bits/byte; pool does not fall below the configured minimum; no blocking or error raised. ✔️ | Linux (kernel ≥ 5.6) / Windows Server 2019+; Java 17 or Python 3.11; NIST SP 800-22 test suite installed. ✔️ |
| TC-02 | Generated keys are statistically unique across bulk generation | Key generation service is running; 10,000 requests are feasible in the test window. ✔️ | (1) Call the key-generation API 10,000 times with identical parameters. (2) Store all returned keys. (3) Check for duplicates using a hash-set comparison. ✔️ | Zero duplicate keys across 10,000 calls. ⚠️ | Staging; k6 or Locust for load generation; in-memory deduplication script. ✔️ |
| TC-03 | Key length matches requested algorithm parameters | Service supports RSA-2048/4096, AES-128/256, ECDSA P-256. ✔️ | (1) Request one key per algorithm/size combination. (2) Inspect the raw bit-length of each key. (3) Validate against spec. ✔️ | Each key matches its specified bit-length with no truncation or padding artefacts. ✔️ | Dev/staging; OpenSSL 3.x; JUnit or pytest. ✔️ |
| TC-04 | System blocks key generation when entropy pool is critically low | Entropy pool monitoring enabled; controlled test environment allows entropy depletion. ✔️ | (1) Configure entropy threshold to a testable value. (2) Reduce pool to below threshold. (3) Attempt key generation. (4) Observe behaviour and logs. ✔️ | Generation is refused until entropy is replenished; appropriate error is logged; no weak key is issued. ✔️ | Isolated VM; /proc/sys/kernel/random/entropy_avail observable; security logging active. ✔️ |
| TC-05 | Keys pass NIST SP 800-22 randomness test battery | 1,000+ keys (≥ 256 bits each) available; NIST SP 800-22 suite installed. ✔️ | (1) Generate 1,000 keys and concatenate bit strings into a test file. (2) Run the full NIST SP 800-22 battery. (3) Record pass/fail per sub-test. ✔️ | All 15 statistical tests pass at α = 0.01; no systematic bias detected. ✔️ | Linux test server; NIST SP 800-22 Rev 1a; Python 3.11. ✔️ |
| TC-06 | System does not use a predictable or timestamp-based seed | Source-code review or black-box API access; SAST scanner configured. ✔️ | (1) Scan codebase for Math.random(), time(), or weak seeds. (2) Generate two keys within the same millisecond. (3) Confirm keys differ and show no temporal pattern. ✔️ | No weak seed found; keys generated within the same millisecond are entirely distinct; SAST reports zero entropy-related findings. ✔️ | CI pipeline with Semgrep or Checkmarx; high-resolution clock for rapid successive calls. ✔️ |
| TC-07 | Key material is not exposed in logs, errors, or API responses | Logging infrastructure active; a test key is generated and its value is known. ✔️ | (1) Generate a key and record its value. (2) Search logs, error reports, and debug output for the key material. (3) Trigger an error mid-generation and inspect the response. ⚠️ | Key material does not appear in any log, stack trace, or error message; only the key identifier is referenced elsewhere. ⚠️ | Staging with ELK stack; log search tooling; error injection framework. ✔️ |
| TC-08 | Key generation latency stays within SLA under normal load | SLA baselines defined (e.g. p95 ≤ 200 ms); load tooling configured. ✔️ | (1) Send 100 concurrent AES-256 key generation requests. (2) Measure per-request response time. (3) Calculate p50, p95, and p99 latencies. ✔️ | p50 ≤ 50 ms, p95 ≤ 200 ms, p99 ≤ 500 ms; no timeouts; entropy quality maintained throughout. ⚠️ | Staging with representative hardware; k6 or JMeter; APM tooling (Datadog / New Relic). ✔️ |
| TC-09 | Entropy pool replenishment keeps pace with sustained demand | Hardware or software entropy source available; real-time pool monitoring exposed. ✔️ | (1) Run sustained key generation at peak rate for 5 minutes. (2) Monitor pool depth every 10 seconds. (3) Stop generation and observe recovery time. ✔️ | Pool depth never falls below minimum safe threshold; pool recovers to ≥ 50% within 30 seconds of load stopping. ✔️ | Production-equivalent; HAVEGED or RDRAND configured; Prometheus + Grafana. ✔️ |
| TC-10 | Graceful failure when entropy source is unavailable | Ability to simulate entropy source failure (disable /dev/urandom or HWRNG in a test VM). ✔️ | (1) Disable or mock-fail the entropy source. (2) Request key generation. (3) Observe response and logs. ✔️ | System returns a safe error (e.g. HTTP 503); no key is generated; internal logs record the failure in detail. ✔️ | Isolated VM; entropy source controllable via test harness; security and application logs accessible. ✔️ |
| TC-11 | Request for an unsupported or insecure key size is rejected | Key generation API running; supported key sizes documented. ✔️ | (1) Submit a request for an insecure key size (e.g. RSA-512). (2) Submit a request with a zero or negative key length. (3) Observe API responses. ✔️ | Both requests rejected with HTTP 400; no key material generated; error message references valid parameter range. ✔️ | Dev/staging; Postman or curl; input validation regression tests. ✔️ |
