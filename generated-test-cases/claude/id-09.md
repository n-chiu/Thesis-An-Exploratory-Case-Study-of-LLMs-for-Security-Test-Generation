# ID 09

| TC ID & Title | Preconditions | Test Steps | Expected Results | Priority | Environment |
|---------------|--------------|------------|------------------|----------|-------------|
| TC-SS-001 Session token uniqueness across logins | Valid user account exists. User is logged out. | 1. Log in with valid credentials — capture session token A. 2. Log out. 3. Log in again — capture session token B. 4. Compare tokens B and A. | Tokens A and B are different. Each login produces a cryptographically unique token. No token is reused. | High | Staging (HTTPS) / Browser + API |
| TC-SS-002 Session fixation prevention | App is accessible; attacker can create a URL with a known session ID. | 1. Attacker sets a known session ID in a cookie/URL parameter before login. 2. Victim visits the crafted URL and authenticates. 3. Observe the post-login session ID. | App invalidates the pre-login session ID and issues a brand-new token upon successful authentication. The fixed ID is rejected. | High | Staging / Proxy tool (e.g., Burp) |
| TC-SS-003 Session expiry after inactivity | User is authenticated. Inactivity timeout is configured (e.g., 15 min). | 1. Log in and note the session token. 2. Leave the session idle beyond the configured timeout. 3. Attempt to perform an authenticated action. | Session is invalidated server-side. User is redirected to the login page. The old token is rejected with a 401/403 response. | High | Staging / Configurable timeout |
| TC-SS-004 Session termination on logout | User is authenticated with an active session token. | 1. Log in and capture the session token. 2. Perform a logout action. 3. Replay the captured token in a new request. | Server-side session is fully destroyed on logout. Replaying the old token returns 401. No sensitive data is accessible. | High | Staging / API client |
| TC-SS-005 Secure and HttpOnly cookie flags | App uses cookies for session management. DevTools or proxy is available. | 1. Log in and inspect the Set-Cookie response header for the session cookie. 2. Check for Secure, HttpOnly, and SameSite attributes. 3. Attempt to access the cookie via document.cookie in browser console. | Cookie has Secure, HttpOnly, and SameSite=Strict (or Lax) flags set. The cookie is not accessible via JavaScript. It is only sent over HTTPS. | High | HTTPS only / Chrome DevTools / Burp Suite |
| TC-SS-006 Resistance to session token brute force | App is running. Automated testing tool (e.g., Burp Intruder) is available. | 1. Observe token format and length. 2. Attempt to brute-force or enumerate valid session tokens using automated requests. 3. Monitor server response. | Tokens have sufficient entropy (>128 bits) making brute force computationally infeasible. Rate limiting or account lockout is triggered; frequency of space is logged and alerted. | High | Staging / Rate-limit enabled / Burp Intruder |
| TC-SS-007 Concurrent session control | Valid user account exists. Two separate browsers or devices are available. | 1. Log in from Browser A — capture session token A. 2. Log in from Browser B with same credentials — capture session token B. 3. Attempt an authenticated action using token A. | Depending on policy: either Token A is invalidated (single-session enforcement) OR both sessions remain active but tracked independently. User may receive an alert on concurrent login. | Medium | Staging / Two browsers/devices |
| TC-SS-008 Session tokens not exposed in URL or logs | App is running. Access to server logs and browser URL bar is available. | 1. Log in and navigate through the app. 2. Inspect the browser URL bar for session tokens in query parameters. 3. Check Referrer headers in outbound requests. 4. Inspect server access logs for token leakage. | Session token never appears in the URL, browser history, or Referrer headers. Server logs do not contain session token values. | High | Staging / Proxy tool / Server log access |
| TC-SS-009 Session binding to client context (IP/User-Agent) | User is authenticated. A second device or proxy tool is available to simulate a different client. | 1. Log in from Client A and capture the session token. 2. Replay the stolen token from Client B (different IP or User-Agent). 3. Observe whether the request is accepted or rejected. | The application detects the change in context and either rejects the token with 401, prompts for re-authentication, or flags a security event. No access is granted in the hijacked context. | Medium | Staging / VPN / Proxy / Burp Suite |
| TC-SS-010 Absolute session lifetime enforcement | User is authenticated. Maximum absolute session duration is configured (e.g., 8 hours). | 1. Log in and record session creation time. 2. Keep the session active by making continuous requests. 3. After the absolute timeout has elapsed, perform an authenticated action. | The session is forcibly invalidated server-side after the absolute maximum lifetime, regardless of activity. User is redirected to login. Re-authentication is required to continue. | Medium | Staging / Configurable max-age / Advanced timer |
