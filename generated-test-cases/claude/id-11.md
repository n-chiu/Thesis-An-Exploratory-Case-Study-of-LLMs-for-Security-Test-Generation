# ID 11

| ID | Test Case Title / Scenario | Preconditions | Test Steps | Expected Results | Environment |
|----|---------------------------|--------------|------------|------------------|-------------|
| TC-001 | Valid input accepted for a required text field | Application is running. User is on a form containing a required text field (e.g. "Full Name"). | 1. Navigate to the form page. 2. Enter a valid string (e.g. "Jane Smith") in the required text field. 3. Submit the form. | Form submits successfully. No validation error is displayed. Data is saved/processed as expected. | Web (Chrome 124+, Firefox 125+), Windows 11 / macOS 14 |
| TC-002 | Required field left empty on submission | Application is running. User is on a form with at least one required field. | 1. Navigate to the form page. 2. Leave the required field blank. 3. Click the Submit button. | Form does not submit. An inline error message (e.g. "This field is required") appears adjacent to the empty field. Focus moves to the first invalid field. | Web (Chrome, Firefox, Safari, Edge), Windows 11 / macOS 14 / iOS 17 |
| TC-003 | Invalid email format rejected | Application is running. A form with an email input field is accessible. | 1. Navigate to the form. 2. Enter a malformed email (e.g. "user@", "plaintext", "user@@domain.com"). 3. Click Submit or move focus away from the field. | An error message stating the email is invalid is displayed (e.g. "Please enter a valid email address"). The form is not submitted. | Web (all major browsers), Windows 11 / macOS 14 |
| TC-004 | Numeric field rejects non-numeric characters | A form contains a field that accepts numbers only (e.g. "Age" or "Quantity"). | 1. Navigate to the numeric input field. 2. Type alphabetic or special characters (e.g. "abc", "!@#"). 3. Attempt to submit the form. | Non-numeric characters are either rejected at the point of entry or flagged with an error on submit. The form does not proceed with invalid data. | Web (Chrome, Firefox, Edge), Windows 11 |
| TC-005 | Input exceeding maximum character limit is rejected | A text field has a defined maximum character limit (e.g. 255 characters). | 1. Navigate to the constrained text field. 2. Paste or type a string that exceeds the allowed limit. 3. Attempt to submit the form. | Input is truncated at the defined limit or an error is shown (e.g. "Maximum 255 characters allowed"). The oversized value is not persisted. | Web (all major browsers), Windows 11 / macOS 14 |
| TC-006 | Password field enforces complexity rules | A registration or password-change form is available with defined complexity rules (min 8 chars, uppercase, number, special character). | 1. Navigate to the password field. 2. Enter a weak password (e.g. "password"). 3. Submit or move focus away. | A descriptive error lists the unmet requirements (e.g. "Must include at least one number and one special character"). The form does not submit. | Web (Chrome, Firefox, Edge, Safari), Windows 11 / macOS 14 / iOS 17 |
| TC-007 | Date field rejects logically impossible dates | A form contains a date input field. | 1. Enter an invalid date such as "31/02/2024" (Feb 31) or "00/00/0000". 2. Attempt to submit the form. | The application displays a date-validation error (e.g. "Please enter a valid date"). The invalid date is not accepted or stored. | Web (all major browsers), Windows 11 |
| TC-008 | Input at exact boundary values is handled correctly | A numeric field has defined min/max constraints (e.g. 1–100). | 1. Enter the minimum boundary value (e.g. 1). 2. Submit and verify. 3. Repeat with the maximum boundary value (e.g. 100). 4. Repeat with values just outside bounds (0 and 101). | Boundary values (1 and 100) are accepted and processed. Values outside the boundary (0 and 101) are rejected with an appropriate message (e.g. "Value must be between 1 and 100"). | Web (Chrome, Firefox), Windows 11 |
| TC-009 | Input containing SQL injection patterns is safely handled | Application is running with a database back-end. A text input field is present. | 1. Enter a SQL injection string (e.g. ' OR '1'='1) into a text field. 2. Submit the form. | The application sanitises or parameterises the input. No SQL error is exposed to the user. The injected string is not executed. A generic validation or error message may be shown. | Web (Chrome), Windows 11; back-end staging environment with test DB |
| TC-010 | Input containing XSS script tags is safely handled | Application is running. A text input field that renders user content is available. | 1. Enter <script>alert('XSS')</script> into a text field. 2. Submit the form. 3. Navigate to the page where the value is displayed. | The script tag is escaped or stripped. No alert box appears. The raw text may be displayed as plain text or omitted, but no script executes. | Web (Chrome, Firefox), Windows 11; staging environment |
| TC-011 | Whitespace-only input is treated as empty | A form with a required text field is available. | 1. Enter only spaces or tabs into a required field. 2. Submit the form. | The application trims whitespace and treats the field as empty. A "required" validation error is shown; the form does not submit. | Web (all major browsers), Windows 11 / macOS 14 |
| TC-012 | Pasted content from clipboard is validated on submit | A form with a validated input field is available. | 1. Copy a valid value to the clipboard. 2. Paste it into the input field using Ctrl+V / Cmd+V. 3. Submit the form. | Pasted valid content passes validation and the form submits successfully. Validation rules apply equally to typed and pasted input. | Web (Chrome, Firefox, Edge, Safari), Windows 11 / macOS 14 |
| TC-013 | File upload validates allowed file type and size | A file upload input accepts only PDF files up to 5 MB. | 1. Attempt to upload a .exe file. 2. Attempt to upload a PDF larger than 5 MB. 3. Attempt to upload a valid PDF under 5 MB. | The .exe upload is rejected with a type error. The oversized PDF is rejected with a size error. The valid PDF is accepted and uploaded successfully. | Web (Chrome, Firefox), Windows 11; back-end staging environment |
| TC-014 | Validation error messages are clear and accessible | A form with multiple field types is accessible. A screen reader (NVDA or VoiceOver) is available. | 1. Submit the form with several invalid fields. 2. Inspect the error messages visually. 3. Use a screen reader to navigate to the error messages. | Each error message is descriptive, actionable, and associated with the relevant field via aria-describedby or equivalent. Screen reader announces errors correctly. Error text has sufficient colour contrast (WCAG AA). | Web (Chrome + NVDA on Windows 11; Safari + VoiceOver on macOS 14) |
