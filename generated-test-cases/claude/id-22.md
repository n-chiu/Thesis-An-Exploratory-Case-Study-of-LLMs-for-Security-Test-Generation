# ID 22

| Test Case Title / Scenario | Preconditions | Test Steps | Expected Results | Environment Specification |
|---|---|---|---|---|
| (1) SQL Injection Prevention | App connected to a database; user has access to an input field. | 1. Navigate to a login or search field. 2. Enter SQL payload (e.g., ' OR '1'='1). 3. Submit the form. | Application rejects or sanitizes the input; no unauthorized data is returned; error is logged securely. | Staging env with WAF enabled; OWASP ZAP or SQLMap available. |
| (2) Cross-Site Scripting (XSS) Prevention | Web application is running; user can submit text input. | 1. Navigate to a comment or form field. 2. Enter script payload (e.g., <script>alert('XSS')</script>). 3. Submit and reload page. | Script is not executed; input is escaped/sanitized and displayed as plain text. | Browser-based testing env; Chrome/Firefox latest; Burp Suite available. |
| (3) Cross-Site Request Forgery (CSRF) Protection | Authenticated user session is active. | 1. Capture a legitimate POST request. 2. Craft a forged request from a different origin. 3. Attempt to submit the forged request. | Request is rejected with a 403 error; CSRF token validation prevents the action from completing. | Staging env; Burp Suite or OWASP ZAP for request interception. |
| (4) File Upload Validation | File upload feature is accessible to the user. | 1. Attempt to upload a file with a disallowed extension (e.g., .exe, .php). 2. Attempt to upload an oversized file. 3. Attempt to upload a file with a spoofed MIME type. | All malicious or invalid files are rejected with appropriate error messages; no file is stored on the server. | Staging env; access to file system or cloud storage logs. |
| (5) Command Injection Prevention | Application accepts user input that interacts with server-side processes. | 1. Enter OS command payload into an input field (e.g., ; ls -la or && whoami). 2. Submit the form. | Input is sanitized; no command is executed on the server; error is logged. | Linux-based staging server; application logs accessible. |
| (6) XML External Entity (XXE) Injection Prevention | Application accepts XML input (e.g., via API or file upload). | 1. Craft an XML payload with an external entity reference. 2. Submit the payload to the application endpoint. | External entity is not processed; application returns an error or ignores the entity; no internal file content is exposed. | API testing env; Postman or Burp Suite available. |
| (7) Insecure Deserialization Prevention | Application deserializes data from user input or cookies. | 1. Intercept a serialized object in transit. 2. Tamper with the payload to include malicious instructions. 3. Send the modified payload to the application. | Application rejects or fails safely on the tampered object; no code execution occurs; event is logged. | Staging env; Burp Suite for request interception; Java/.NET environment as applicable. |
| (8) Path Traversal Prevention | Application allows file access or download functionality. | 1. Input a path traversal string into a file parameter (e.g., ../../etc/passwd). 2. Submit the request. | Application rejects the request; no unauthorized file content is returned; error is logged. | Linux-based staging server; access to server logs. |
| (9) Input Length and Format Validation | Application has form fields with defined input constraints. | 1. Enter input exceeding the maximum allowed length. 2. Enter input with invalid characters/format (e.g., HTML tags in a name field). 3. Submit each form. | Inputs are rejected or truncated; validation error messages are displayed; no crash or data leak occurs. | Staging env with standard browser; dev tools enabled. |
| (10) Sensitive Data Exposure via Error Messages | Application generates error messages on invalid input. | 1. Submit malformed input to trigger a server error. 2. Observe error messages returned to the user. | Error messages do not expose stack traces, database schemas, file paths, or internal logic; generic user-facing message is shown. | Staging env; browser dev tools and network inspector. |
| (11) Authentication Bypass via Manipulated Input | Login form is accessible. | 1. Attempt login with blank credentials. 2. Submit encoded or null-byte payloads. 3. Attempt to manipulate hidden form fields. | All bypass attempts are rejected; user is not authenticated; failed attempts are logged and may trigger account lockout. | Staging env; Burp Suite or browser dev tools. |
| (12) HTTP Header Injection Prevention | Application reflects user input in HTTP response headers. | 1. Submit input containing carriage return/line feed characters (e.g., \r\n). 2. Observe the HTTP response headers. | Injected characters are stripped or encoded; no additional headers are injected into the response. | Staging env; Burp Suite for header inspection. |
