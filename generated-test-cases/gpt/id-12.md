|----------------------------|---------------|------------|------------------|--------------------------|
| (1) Verify authorized customer can view assigned application secret keys | Customer account exists; customer is active; application secret key is available in secure secret management system; customer has correct role/permission | 1. Log in as authorized customer.<br>2. Navigate to secret key management section.<br>3. Select assigned application.<br>4. Request to view secret key. | Customer can access only the secret keys assigned to their application/account. Secret is displayed or revealed securely according to policy. Access event is logged. | QA environment with IAM/SSO enabled, secure secret vault integrated, audit logging enabled |
| (2) Verify unauthorized customer cannot access secret keys | Customer account exists without secret access permission | 1. Log in as unauthorized customer.<br>2. Navigate to secret key management section.<br>3. Attempt to access/reveal a secret key. | Access is denied with appropriate error message. Secret value is not exposed. Unauthorized attempt is logged. | QA environment with RBAC enabled, secure secret vault integrated, audit logging enabled |
| (3) Verify customer can access only their own application secret keys | Multiple customer accounts and applications exist with different assigned secrets | 1. Log in as Customer A.<br>2. Navigate to secrets page.<br>3. Attempt to view Customer B’s application secret key directly or via URL/API manipulation. | Customer A can only see secrets for their own assigned applications. Other customers’ secrets are hidden or denied. Attempt is logged. | QA environment with tenant isolation, RBAC enabled, audit logging enabled |
| (4) Verify secret keys are masked by default | Authorized customer exists with access to a secret | 1. Log in as authorized customer.<br>2. Open secret key management page. | Secret keys are masked by default. No full secret is visible unless explicitly revealed through allowed action. | QA environment, UI masking enabled, browser-based access |
| (5) Verify secure reveal of secret key after explicit user action | Authorized customer exists; secret masking is enabled | 1. Log in as authorized customer.<br>2. Navigate to secret key page.<br>3. Click “Reveal” or equivalent action.<br>4. Complete any required secondary verification if configured. | Secret is revealed only after explicit action and only to authorized user. Reveal action is logged. | QA environment with MFA or step-up authentication enabled if applicable |
| (6) Verify secret key is not exposed in browser page source or network responses beyond intended secure channel | Authorized customer exists; access to browser developer tools | 1. Log in as authorized customer.<br>2. Open secret key page without revealing the secret.<br>3. Inspect page source, DOM, and network traffic.<br>4. Reveal secret and inspect traffic again. | Secret is not present in page source or hidden DOM before reveal. Secret is transmitted only when required and via secure encrypted channel. No unnecessary exposure occurs. | QA environment over HTTPS/TLS, browser dev tools available |
| (7) Verify secret key access requires authentication | Secret management feature is deployed; no active user session | 1. Open secret management URL directly without logging in.<br>2. Attempt to access secret details endpoint/API. | Unauthenticated user is redirected to login or receives unauthorized response. No secret information is exposed. | QA environment with authentication enforced, web/API endpoints exposed |
| (8) Verify session timeout prevents continued secret access | Authorized customer session exists; timeout policy configured | 1. Log in as authorized customer.<br>2. Navigate to secret key page.<br>3. Remain inactive until session expires.<br>4. Attempt to reveal or copy secret key. | User is required to re-authenticate after timeout. Secret is not accessible from expired session. | QA environment with session timeout policy enabled |
| (9) Verify audit log is created for secret key access | Authorized customer exists; audit logging enabled | 1. Log in as authorized customer.<br>2. View/reveal/copy a secret key.<br>3. Access audit logs as admin or auditor. | Audit log captures user ID, timestamp, action type, application/secret identifier, and outcome. Sensitive value itself is not logged. | QA environment with centralized logging/audit tools enabled |
| (10) Verify secret values are never written to application logs | Authorized customer exists; application logging enabled | 1. Log in as authorized customer.<br>2. Access/reveal/copy secret key.<br>3. Review application, API, and system logs. | Logs contain metadata about the action but never contain full secret values. | QA environment with log aggregation enabled |
| (11) Verify customer can copy secret key securely if feature is supported | Authorized customer exists; copy feature enabled | 1. Log in as authorized customer.<br>2. Navigate to secret key page.<br>3. Reveal secret.<br>4. Use copy-to-clipboard action. | Secret is copied successfully only for authorized user. UI confirms copy action without exposing extra information. Copy event is logged if required. | QA environment, supported browser, clipboard permissions enabled |
| (12) Verify secret access over API follows authorization rules | API access is enabled; valid API token for authorized and unauthorized customers exists | 1. Call secret retrieval API using authorized customer token.<br>2. Call same API using unauthorized customer token.<br>3. Call API without token. | Authorized request succeeds per policy. Unauthorized and unauthenticated requests fail with proper status codes. No unintended secret disclosure occurs. | QA environment with API gateway, OAuth/API token auth enabled |
| (13) Verify rotated secret key is updated and old key is no longer accessible if policy requires | Secret rotation capability exists; old and new versions available | 1. Log in as authorized customer.<br>2. View current secret key.<br>3. Trigger or wait for rotation.<br>4. Refresh/retrieve secret again.<br>5. Attempt to access deprecated version if restricted. | Customer sees current valid secret version. Old/revoked secret is inaccessible if policy prohibits historical access. Rotation event is logged. | QA environment with secret rotation enabled, versioning policy configured |
| (14) Verify revoked or suspended customer cannot access secret keys | Customer account exists and is then suspended/revoked | 1. Suspend or deactivate customer account.<br>2. Attempt login or secret access using existing session/token. | Access is denied immediately or at next validation point. Existing sessions/tokens are invalidated per policy. | QA environment with account lifecycle management enabled |
| (15) Verify MFA or step-up authentication is enforced for sensitive secret access if required | Authorized customer exists; MFA policy configured for secret viewing | 1. Log in as authorized customer without completing MFA.<br>2. Attempt to reveal secret.<br>3. Complete MFA.<br>4. Retry reveal. | Reveal is blocked until MFA or secondary verification is completed. After successful verification, access is granted. | QA environment with MFA provider integrated |
| (16) Verify secret key access is restricted by environment or application scope | Customer has access to one environment/application only; multiple environments exist (Dev/Test/Prod) | 1. Log in as authorized customer.<br>2. Access allowed environment/application secret.<br>3. Attempt to access secret from restricted environment/application | Customer can access only permitted environment/application secrets. Restricted scope is denied. | QA environment with scoped permissions by application/environment |
| (17) Verify concurrent access does not expose wrong secret to wrong customer | Two customer accounts and different secrets exist | 1. Log in as Customer A and Customer B in separate sessions.<br>2. Access secret pages simultaneously.<br>3. Reveal secrets in both sessions. | Each customer sees only their own secret. No session leakage, caching issue, or cross-tenant data mix occurs. | QA environment, multi-session browser or separate browsers, tenant isolation enabled |
| (18) Verify cached pages do not expose secret keys after logout | Authorized customer exists; browser caching enabled | 1. Log in and reveal a secret key.<br>2. Log out.<br>3. Use browser back button or cached page revisit. | Secret is not visible after logout via cached page. User is redirected to login or shown sanitized page. | QA environment with browser access, cache-control headers configured |
| (19) Verify error handling does not expose secret information | Authorized customer exists; fault injection or backend error simulation available | 1. Trigger secret retrieval during backend/service failure.<br>2. Observe UI/API error response. | Error message is generic and helpful but does not include secret values, internal stack traces, or sensitive implementation details. | QA environment with failure simulation capability |
| (20) Verify secret access complies with encryption in transit and at rest requirements | Secret storage system configured with encryption | 1. Review configuration or execute retrieval flow.<br>2. Validate HTTPS/TLS in transit.<br>3. Validate vault/storage encryption settings at rest. | Secret is encrypted in transit and stored encrypted at rest according to security requirements. | QA or staging environment with TLS enabled and secure vault/storage configuration |
