# ID 19

| Test Case Title/Scenario | Preconditions | Test Steps | Expected Results | Environment Specification |
|----------------------------|--------------|------------|------------------|--------------------------|
| TC-01: Compromised component cannot access other application components directly | Application is deployed in an environment with network segmentation or service-to-service access controls enabled. Vulnerable component and at least two non-vulnerable components are running. | 1. Identify the vulnerable component.<br>2. Simulate malicious activity from the vulnerable component, such as unauthorized API calls to another internal component.<br>3. Attempt communication over restricted ports/protocols.<br>4. Review network and application logs. | Unauthorized east-west traffic is blocked. Other application components remain operational and unaffected. Security events are logged. | Integration or staging environment mirroring production network policies, service mesh, firewall rules, and internal routing. |
| TC-02: Compromised component cannot read or modify another component’s data store | Each component uses separate credentials, roles, or database schemas. Access control is enforced. | 1. Simulate compromise of the vulnerable component.<br>2. Use its runtime identity or credentials to query another component’s database or storage.<br>3. Attempt read, write, update, and delete operations.<br>4. Check audit logs. | Access is denied for unauthorized data operations. No cross-component data is exposed or modified. Audit trail shows denied access attempts. | Staging environment with production-like IAM roles, database permissions, and audit logging enabled. |
| TC-03: Compromised component cannot escalate privileges | Role-based access control is configured. Vulnerable component runs with least privilege. | 1. Simulate malicious code execution in the vulnerable component.<br>2. Attempt to assume elevated roles, use admin APIs, or access privileged system commands.<br>3. Review authorization and system logs. | Privilege escalation attempts fail. Component remains restricted to its assigned role. No privileged actions are executed. | Test environment with production-equivalent IAM, container runtime permissions, and admin APIs available for validation. |
| TC-04: Excessive resource consumption in one component does not crash the whole application | Resource quotas, limits, or container constraints are configured. Monitoring is enabled. | 1. Generate CPU and memory spikes in the vulnerable component.<br>2. Observe neighbouring components and shared infrastructure.<br>3. Attempt normal user transactions through unaffected application paths.<br>4. Review alerts and metrics. | Resource exhaustion is contained to the vulnerable component. Other components remain responsive within agreed thresholds. Monitoring raises alerts. | Performance or resilience test environment with autoscaling, quotas, and observability stack enabled. |
| TC-05: Failure of vulnerable component is isolated through graceful degradation | Application supports fallback behaviour, timeout handling, or circuit breakers. | 1. Force the vulnerable component into an unstable state.<br>2. Send requests that depend on it and requests that do not.<br>3. Observe UI/API behaviour and downstream services. | Dependent features fail gracefully with controlled error messages or degraded functionality. Unrelated features continue working normally. No cascading failure occurs. | Staging environment with circuit breaker, retry, timeout, and fallback configurations aligned to production. |
| TC-06: Malicious outbound traffic from a compromised component is blocked | Egress filtering or outbound proxy rules are configured. Known blocked external destinations are available for test. | 1. Simulate malicious code in the vulnerable component.<br>2. Attempt outbound calls to unauthorized external IPs/domains.<br>3. Attempt DNS resolution and data transfer.<br>4. Review egress and security logs. | Unauthorized outbound traffic is blocked. No data leaves the environment through unauthorized channels. Events are logged and alertable. | Security test environment with production-like egress controls, DNS policies, and network logging. |
| TC-07: Session and authentication boundaries prevent impact on other users or components | Authentication, token scoping, and session isolation are configured. Test users with different roles exist. | 1. Compromise the vulnerable component handling a user session.<br>2. Attempt to reuse, forge, or broaden session tokens.<br>3. Attempt actions as another user or against another component.<br>4. Validate token scope and access checks. | Tokens cannot be expanded beyond intended scope. Other users’ sessions and component boundaries remain protected. Unauthorized actions are denied. | Test environment with production-like identity provider, token signing, and session management settings. |
| TC-08: Compromised component cannot access secrets belonging to other components | Secrets are stored in a vault or isolated secret store. Components receive only their own secrets. | 1. Simulate compromise of the vulnerable component.<br>2. Attempt to enumerate environment variables, mounted secrets, or vault paths.<br>3. Try retrieving secrets assigned to other components.<br>4. Review secret access logs. | Only secrets assigned to the compromised component are accessible. Other components’ secrets remain inaccessible. Unauthorized secret access attempts are logged. | Staging environment with production-like secrets manager, secret injection method, and audit logging enabled. |
| TC-09: Lateral movement through shared filesystem or runtime is prevented | Components are isolated by container, namespace, host policy, or filesystem permissions. | 1. Simulate malicious code execution in the vulnerable component.<br>2. Attempt to access shared volumes, host filesystem paths, or neighbouring container resources.<br>3. Attempt to plant files or scripts for another component to execute.<br>4. Review runtime and host logs. | Access outside the component’s authorized boundary is denied. No persistence or tampering occurs in neighbouring components. | Containerized staging environment with production-equivalent runtime hardening, volume mappings, and host policies. |
| TC-10: Message queue or event bus abuse from a compromised component is contained | Event topics/queues enforce publish and subscribe authorization. Non-vulnerable components consume separate topics or scoped messages. | 1. Simulate compromise of the vulnerable component.<br>2. Attempt to publish unauthorized messages to other components’ topics.<br>3. Attempt to consume messages from unauthorized subscriptions.<br>4. Observe downstream processing behaviour. | Unauthorized publish/subscribe actions are denied. Other components do not process malicious cross-boundary messages. Logs capture denied attempts. | Integration environment with production-like message broker, topic ACLs, and consumer authorization enabled. |
| TC-11: Malicious input handled by one component does not corrupt shared application state | Shared caches, state stores, or configuration repositories are protected by validation and scoped access. | 1. Send malformed or malicious payloads to the vulnerable component.<br>2. Attempt to poison shared cache entries or shared state objects.<br>3. Access unaffected components that rely on shared state.<br>4. Validate application consistency. | Malicious input is rejected or isolated. Shared state remains valid. Other components continue functioning correctly without corrupted behaviour. | Integration environment with shared cache/state services configured as in production. |
| TC-12: Security monitoring detects and contains suspicious behaviour in the vulnerable component | SIEM, alerts, runtime detection, and optional auto-containment are configured. | 1. Simulate indicators of compromise such as unusual process execution, excessive failed access attempts, or blocked lateral movement.<br>2. Observe monitoring tools and any automated containment actions.<br>3. Verify impact on other application components. | Suspicious behaviour is detected promptly. Alerts are raised. If enabled, containment action isolates the component without disrupting the rest of the application. | Security monitoring test environment with production-equivalent alert rules, log forwarding, and response automation. |
| TC-13: Restart or redeploy of compromised component does not affect unrelated components | Application supports rolling restart, self-healing, or pod replacement. | 1. Mark the vulnerable component as unhealthy or terminate it.<br>2. Trigger restart/redeployment.<br>3. Continue exercising unrelated features during recovery.<br>4. Review availability and error rates. | Only the vulnerable component is restarted or replaced. Unrelated components remain available. Recovery completes without application-wide outage. | Orchestrated staging environment with production-like deployment strategy and health checks. |
| TC-14: Denial-of-service against one public endpoint does not degrade unrelated endpoints | Rate limiting, WAF, and per-endpoint throttling are configured. Multiple public endpoints exist. | 1. Flood requests to the endpoint served by the vulnerable component.<br>2. Simultaneously send normal traffic to unrelated endpoints.<br>3. Measure latency, throughput, and error rates.<br>4. Review rate-limit and WAF logs. | Attack traffic is throttled or blocked. Unrelated endpoints remain within acceptable performance limits. Security controls log the event. | Performance/security test environment with production-like API gateway, WAF, and rate-limiting configuration. |
| TC-15: Compromised component cannot alter global configuration used by the rest of the application | Global configuration changes require separate privileged access and approval path. | 1. Simulate malicious activity in the vulnerable component.<br>2. Attempt to modify feature flags, config maps, central settings, or service discovery records.<br>3. Validate application behaviour and audit logs. | Unauthorized configuration changes are denied. Rest of application continues operating with trusted configuration. Audit logs record attempted tampering. | Staging environment with production-like configuration management, RBAC, and change auditing. |
